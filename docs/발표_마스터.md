## S1. 프로젝트 정체성과 가치 명제

### 한 줄 정의 (발표 첫 30초용)
> **제주 설화·민담 공공데이터를 활용해, 실제 관광객 검증 코스 위에 AI 동행자가 설화를 얹어주는 iOS 앱.**
> 계획·여행·기록이 한 앱에서 끊김 없이 이어진다.

### 문제 정의 (왜 만들었나)
1. **관광객 입장의 결핍**: 제주는 연간 1,500만 명이 방문하지만, 현재 관광 앱은 맛집·숙박 정보 중심이라 제주의 문화적 깊이를 경험할 통로가 없다.
2. **공공데이터 활용 부재**: 제주특별자치도가 개방한 설화 182건 + 민담 323건 = **총 505건의 무형 문화유산 데이터**가 활용 사례 없이 잠들어 있다.

→ 두 문제를 한 번에 풀어냄.

### 핵심 가치 명제 (한 줄 슬로건)
> **"실제 검증된 여행 코스 위에 AI가 설화를 얹는다."**

왜 강력한가?
- AI가 코스를 처음부터 생성하면 비현실적 동선이 나옴 → 실제 관광객 검증 코스가 신뢰성 보장
- AI는 "큐레이션·해석"만 담당, "동선 생성"은 데이터에 위임 → **AI의 약점(환각·비현실성)을 우회한 설계**
- 단순 추천이 아니라 **계획·여행·기록 단일 여정** (대부분 앱은 계획 또는 가이드 중 하나만)
- **5종 페르소나 AI 동행자**가 같은 설화를 다섯 말투로 풀어줌 — 콘텐츠 다양성

### 사용자 흐름 (3단계 정확한 디테일)

#### 계획
홈 → 지역(5지선다: 북부/남부/동부/서부/전체)·설화취향·기간 입력 → 추천 코스 3개 → 미리보기 → 저장

#### 여행 중
저장 코스에서 "탐험 시작" → 지도 + 다음 장소 안내
→ **GPS 도착 감지** 
→ **AI 동행자 오버레이 전체화면 등장** (페르소나 음성으로 자동 인사)
→ **채팅** (페르소나 말투 + 5종 TTS 음성)
→ 채팅 종료 → **리뷰 바텀시트 자동 등장** (5개 태그)
→ 다음 장소로 이동 (반복)

#### 기록
모든 장소 방문 → "탐험 마치기" → Day 요약 (방문 수·설화 수)
→ 일지 생성 (~25초)
→ **GPT-4o 단일 호출로 일지 텍스트 + 민화 이미지 프롬프트 동시 생성**
→ gpt-image-1이 민화 한 장 렌더링
→ 텍스트 일지 + 민화 표시 → SNS 공유

### 부심교수에게 던질 한 학기 결과물 한 문장
> "한 학기 동안 **제주 공공데이터 3종(설화·민담·관광지)을 정제해 벡터 DB와 사전 매핑 테이블로 구축하고, FastAPI 백엔드 + iOS Swift 앱 + Firebase 인증을 모두 직접 구현한 end-to-end 시스템**입니다. App Store 출시와 2026 관광 공모전을 목표로 한 실서비스 수준입니다."

### 발표 첫 1분에 말할 핵심 문장 3줄 (외워야 할 것)
1. **정체성**: "제주 설화·민담 공공데이터를 활용해, 검증된 여행 코스 위에 AI가 설화를 얹어주는 iOS 앱입니다."
2. **문제**: "관광객은 맛집·숙박 정보에만 노출돼 제주 문화의 깊이를 경험할 통로가 없고, 505건의 공공데이터는 활용 사례가 없었습니다."
3. **답**: "실제 관광객 검증 코스 + AI 설화 매핑 + 5종 페르소나 동행자 + 자동 일지·민화 — 계획·여행·기록을 단일 여정으로 묶었습니다."

## S2. 데이터 파이프라인

### 외워야 할 핵심 수치 카드 (★ 최종)
```
📊 출처    : 설화 182 + 민담 323 = 505건  (제주특별자치도 E07/E08 API)
📊 코스    : 9,134개 검증 코스, 146,508 코스-장소  (Visit Jeju)
📊 장소    : 3,284 → 820 관광지 (식당·카페·숙박 제외, -77% 노이즈)
📊 매핑    : 820 관광지 중 524개 ↔ 397 설화·민담 = 6,726건
            (gps_assist 24,221 제외한 정합성 매핑, 한 설화가 평균 17개 장소에 등장)
📊 청크    : 1,749개 적재 (ChromaDB, text-embedding-3-small, 1,536-dim)
            └ 동행자 채팅 검색: 매핑된 설화 청크 (is_mapped=True 필터 적용)
```

### ★ 시스템 전체 정합성 원칙 (한 학기 마지막 의사결정)
> **"근처에 있다 ≠ 그 명소의 설화"** — 비짓제주 관광지와 매핑된 397 설화·민담만 사용자에게 노출. 장소 정합성 100% 보장 위해 두 곳에 동일 설계 패턴 적용:
>
> | 검색 경로 | 메타데이터 | 필터 |
> |---|---|---|
> | ChromaDB 의미 검색 (동행자 채팅) | `is_mapped` | `where={"is_mapped": True}` |
> | 사전 매핑 테이블 (코스 추천) | `source` | `WHERE source != 'gps_assist'` |
>
> 데이터 자체는 보존되어 향후 정책 토글 가능. 일관된 설계 패턴.

### 공공데이터 4종 출처 (3종은 데이터, KTO는 분류/조회)

| # | 출처 | 데이터 | 사용처 |
|---|---|---|---|
| 1 | 제주특별자치도 공공데이터 API (E07) | 설화 182건 | RAG·매핑 |
| 2 | 제주특별자치도 공공데이터 API (E08) | 민담 323건 | RAG·매핑 |
| 3 | 제주관광공사 Visit Jeju 공공데이터 | 여행장소 3,284 + 여행일정 9,134  | 코스 베이스라인 |
| 4 | 한국관광공사 TourAPI (KTO) | 장소 분류 + 사진·관광 정보 | (a) 비짓제주 장소 1단계 분류 (b) 장소 클릭 시 사진/정보 7일 캐시 |

> 공모전 출품 조건상 KTO API 활용 필수

### 설화·민담 → 비짓제주 관광지 매핑 (★ 핵심 정합성 기준)

코드 확인 결과 (`scripts/build_place_folklore_mapping.py`): 설화·민담 본문에서 화이트리스트 기반으로 지명 추출 + 민담은 채록 메타데이터 활용 → 비짓제주 820 관광지와 매칭. **본문에 비짓제주 관광지 지명이 등장하는 설화·민담만 정합성 매핑으로 인정**.

| 데이터 | 정합성 매핑 결과 |
|---|---|
| **설화·민담 505건** | 비짓제주 관광지 등장 매핑 **397건** (78%) |
| **비짓제주 820 관광지** | 매핑 보유 관광지 **524개** (64%) |
| **합계** | **6,726건 정합성 매핑** (한 설화가 평균 17개 장소에 등장) |

#### 화이트리스트 — 제주 행정구역 120개 정답 사전
"제주시", "서귀포시", "한림읍", "성산일출봉", "한라산", "협재", "함덕" 등.
본문에 이 중 하나라도 매칭되면 지명 확정. AI/NER 방식보다 노이즈 0 + 비용 0 + 검증 가능.

#### 매핑되지 않은 108건의 활용
- ChromaDB에 텍스트 임베딩으로 모두 적재됨 (매핑된 397건 청크만 동행자 채팅 검색 대상 — `is_mapped=True` 메타데이터 필터)
- 본문이 매우 짧거나 매칭 가능한 비짓제주 관광지 지명이 없는 경우
- 미매핑 108건은 보존되어 향후 자유 발화 챗봇 등 다른 기능에서 재활용 가능
- 사전 매핑 테이블의 GPS 보조 매핑(8km, 24,221건)은 정합성 우려로 코스 추천에서 `source != 'gps_assist'` 필터로 자동 제외 — 장소 정합성 보장

#### ⚠️ 현재 시스템 정합성 보장 (Q&A 대비)

**원칙**: "근처에 있다 ≠ 그 명소의 설화" — 비짓제주 관광지와 본문에서 직접 매핑된 설화·민담만 사용자에게 노출.

**구현 (코드 확인)** — **[적용 완료]** `is_mapped` 메타데이터 + `where` 필터 적용. 동행자 채팅은 매핑된 설화 청크만 검색:
- `scripts/ingest_to_vector_db.py` — 메타데이터에 `is_mapped` boolean 키 emit (SSoT: 매핑 테이블)
- `services/folklore_search.py` — `collection.query(..., where={"is_mapped": True})` 적용

**Q&A 권장 답**:
> "**ChromaDB 메타데이터에 매핑 플래그(`is_mapped`)를 두고, 동행자 채팅 검색 시 `where={\"is_mapped\": True}` 필터로 비짓제주 관광지와 직접 매핑된 397 설화·민담 청크만 검색 대상으로 제한합니다. 미매핑 108건은 보존되어 향후 다른 기능에서 재활용 가능합니다.**"

### 비짓제주 장소 3단계 필터링 (`scripts/classify_visitjeju_places.py`)

| 단계 | 방법 | 라벨링 수 |
|---|---|---|
| 1단계 | KTO `contenttypeid`(12=관광지/14=문화시설/15=축제/28=레포츠) + `cat1` 매칭 | ~456개 |
| 2단계 | 이름·주소 키워드 사전 (우선순위: 카페 → 식당 → 숙박 → 자연/문화/유적). "갈비·흑돼지·Cafe·오름·해변" 패턴 | ~1,400개 |
| 3단계 | 후처리 — "관광지"로 잘못 분류된 체인 식당 패턴("○○점"+"맛집·주막·정식") 강제 재분류 | ~29개 |

결과: **3,284 → 820 관광지** → 매핑 134,000 → 31,000건 (**-77% 노이즈 제거**)

### 장소-설화 매핑 3가지 방식 조합 (`scripts/build_place_folklore_mapping.py`)

| 방법 | 동작 |
|---|---|
| ① 지명 텍스트 매칭 (주된 방식, 의미적) | 설화 텍스트의 추출 지명이 명소 주소·이름에 포함되면 매핑. 지명 우선순위 점수(읍·면·리 > 시) |
| ② 민담 채록 장소 매칭 | 채록 메타데이터의 조사 장소 → 명소 매칭 |
| ③ GPS 보조 매핑 (8km) | ①②로 못 잡힌 경우 보완. GPS 거리 ≤ 8km면 추가 매핑 |

→ 코스 추천 시점에는 `source != 'gps_assist'` 필터로 ① 지명 텍스트 매칭과 ② 민담 채록 장소 매칭만 활용. ③ GPS 보조는 데이터 보존하되 사용자 노출에서 제외.

→ 단순 거리 계산이 아님. **여행 중 채팅에서는 별도로 ChromaDB 벡터 검색**으로 의미론적 RAG.

### ChromaDB vs SQLite 역할 분리

| DB | 역할 |
|---|---|
| **ChromaDB** | 의미론적 검색만 — "장소명/자유 발화 → 관련 설화 top-3" |
| **SQLite** | 모든 구조화 데이터 — courses, course_places, place_folklore_mapping, place_reviews, tourist_info_cache, folklore_connection_cache(Lv2), folklore_story_cache(Lv3) |

### ChromaDB 정확한 스펙

| 항목 | 값 |
|---|---|
| 컬렉션 | `jeju_folklore_chunks` 단일 |
| 임베딩 모델 | OpenAI `text-embedding-3-small` |
| 벡터 차원 | 1,536차원 |
| 청크 분할 | **900자 + 120자 overlap** (단순 슬라이딩 윈도우) |
| 총 청크 수 | 1,749 청크 |
| 인덱스 | HNSW + **코사인 유사도** |

### Q&A 권장 답 — 핵심 3가지

**Q. "데이터를 어디서 받았어요?"**
> "공공데이터는 3종입니다. 제주특별자치도가 운영하는 설화·민담 API에서 505건, Visit Jeju에서 실제 관광객 검증 코스 9,134개와 장소 3,284개, 그리고 한국관광공사 TourAPI는 장소 분류 보조와 사진·관광정보 조회용으로 사용합니다."

**Q. "왜 3단계 필터링했어요?"**
> "비짓제주 원본을 그대로 두면 '협재해물라면 ↔ 좌수의 횡포' 같은 무의미한 매핑이 13만 건 쌓여 코스에 설화가 의미를 잃습니다. KTO 공공 분류 → 자체 키워드 사전 → 후처리 보정의 3단계 폴백 구조로 매핑 신뢰성을 77% 회복했습니다."

**Q. "장소와 설화는 단순 거리로 연결했나요?"**
> "단순 거리가 아닙니다. 지명 텍스트 매칭이 주된 방식이고, 민담은 채록 장소로 매칭, 그것도 안 되면 GPS 8km로 보완합니다. 그리고 여행 중 채팅에서는 ChromaDB 벡터 검색으로 별도 의미론적 RAG를 수행합니다."

**Q. "사전 매핑 테이블에서 GPS 보조 매핑은 어떻게 처리하나요?"**
> "GPS 보조 매핑은 '근처에 있다 ≠ 그 명소의 설화'라는 정합성 우려가 있어, 사전 매핑 테이블에 source 컬럼을 두고 코스 추천 쿼리에 `source != 'gps_assist'` 필터를 적용합니다. 데이터 자체는 보존되어 향후 정책 변경 시 토글 가능합니다. 이는 ChromaDB의 `is_mapped` 메타데이터 필터와 동일한 설계 패턴으로, 두 검색 경로 모두에서 비짓제주 관광지와 매핑된 397 설화·민담만 노출하는 일관된 장소 정합성을 보장합니다."

## S3. 코스 추천 알고리즘 (★ 정정 사항 — 발표 핵심)

### ⚠️ 중요한 정정
이전 발표자료/보고서에 "GPT-4o가 3개 선별"이라고 적힌 부분은 **부정확**.
실제 코드는 **LLM 호출 0회**, 결정론적 점수 + 무작위 샘플링.

### 1단계 `/course/list` 정확한 흐름 (`agents/course_list_agent.py`)

```
입력: region(지역, 5지선다: 북부/남부/동부/서부/전체), duration_days(기간), category_scores(설화 카테고리 점수)
  ↓
[Step 1] SQLite 후보 추출 — region 값에 따라 분기
  ▶ region == "전체" (제약 없음, 1,090개 모든 코스 후보)
    SELECT * FROM curated_courses
    WHERE duration_days BETWEEN d-1 AND d+1
      AND place_count >= max(3, duration_days)
    ORDER BY composite_score DESC
    LIMIT 50

  ▶ 그 외 (북부/남부/동부/서부 — 4지선다 기존 동작)
    SELECT * FROM curated_courses
    WHERE region IN (요청지역, '전체')
      AND duration_days BETWEEN d-1 AND d+1
      AND place_count >= max(3, duration_days)
    ORDER BY (region 일치 우선), composite_score DESC
    LIMIT 50
  ↓
[Step 2] 각 후보에 가중 점수 _score_course() 계산
  - course_places JOIN place_folklore_mapping (specificity >= 5)
  - 가중치 w = specificity / 5.0
    · spec=5  → w=1.0 (광역 매칭)
    · spec=10 → w=2.0 (정확한 지명, 예: 한라산·성산일출봉)
  - 카테고리별 가중 누적 → 점유율 share = w / Σw
  - score = Σ(사용자_카테고리_점수 × share) × 10
  ↓
[Step 3] score > 0인 풀 만들고 random.sample(pool, 3)
  ↓
출력: 3개 코스 (LLM 호출 0회)
```

### 📌 코스 점수 계산 — 직관적 예시 (발표용)

#### 시나리오
- **사용자 입력**:
  - region: "북부"
  - 1순위 카테고리: 마을 공동체 전승 → **3점**
  - 2순위 카테고리: 초자연 존재담 → **1점**
  - 나머지 3개 카테고리: 0점
  - duration_days: 2

#### Step 1 — region·일수로 후보 50개 추출 (카테고리 안 봄)

```sql
SELECT * FROM curated_courses
WHERE region IN ('북부', '전체')
  AND duration_days BETWEEN 1 AND 3
LIMIT 50
```

#### Step 2 — 각 후보에 점수 계산 (이때 카테고리 사용)

**예시 — 후보 "코스 A"의 매핑 (place_folklore_mapping 조회 결과)**:

```
협재해수욕장 ↔ 「용궁 이야기」       → 해양·어촌 전승
협재해수욕장 ↔ 「각시당본풀이」      → 마을 공동체 전승
한라산      ↔ 「설문대할망 신화」   → 무속신화·신격 전승
한라산      ↔ 「산방산 도체비담」   → 초자연 존재담
                                    (총 4건 매핑)
```

**카테고리별 점유율 계산**:
```
마을 공동체 전승  : 1/4 = 25%
초자연 존재담    : 1/4 = 25%
해양·어촌 전승    : 1/4 = 25%
무속신화·신격 전승 : 1/4 = 25%
생활민담·교훈담   : 0/4 = 0%
```

**사용자 점수와 곱한 합산**:
```
score = (3점 × 25%)  ← 마을 공동체 (사용자 1순위)
      + (1점 × 25%)  ← 초자연 (사용자 2순위)
      + (0점 × 25%)  ← 해양 (사용자 점수 0)
      + (0점 × 25%)  ← 무속신화 (사용자 점수 0)
      + (0점 × 0%)   ← 생활민담
      = 0.75 + 0.25
      = 1.0 × 10 (스케일)
      = 10점
```

#### Step 3 — score > 0인 풀에서 무작위 3개

```
score > 0 인 후보 풀:
  코스 A (10점), 코스 B (5점), 코스 D (8점), 코스 E (12점), ...
                     ↓ random.sample(3)
              선택: 코스 A, D, F (매번 다름)
```

**핵심**: 점수 1·2·3등을 순서대로 주는 게 아니라, **점수 양수 풀에서 매번 무작위 3개**.

### 발표용 한 줄 요약

> **"region + 일수로 후보 50개 → 사용자 카테고리 점수 × 매핑 점유율로 score 계산 → score>0 풀에서 매번 무작위 3개"**

```
입력 region·일수        → SQL 필터       → 50개 후보
                                         ↓
입력 카테고리 점수       → 점수 계산      → score>0 풀
                                         ↓
                                random.sample(3) → 결과
```

### 2단계 `/course/detail` 정확한 흐름 (`agents/course_detail_agent.py`)

```
입력: course_id, category_scores
  ↓
[Step 1] SQLite에서 코스 장소 목록 조회
  ↓
[Step 2] 각 장소에 대해 사전 매핑 테이블 조회
  SELECT * FROM place_folklore_mapping
  WHERE place_name = ?
  ORDER BY specificity DESC
  LIMIT 20
  ↓
[Step 3] 카테고리 점수 + specificity로 재정렬

  ↓
[Step 4] (비활성화) generate_narrative() 호출 안 함

  - 시연 응답 속도/비용 절감 위해 의도적으로 비활성화
  - narrative = "" 반환
  ↓
출력: 코스 + 매핑된 설화 (LLM 호출 0회)
```

### 핵심 의사결정 — 왜 LLM을 안 쓰는가 (부심교수 어필 포인트)

> **"AI는 '값비싼 곳'에만 집중하도록 시스템을 설계했습니다.
> 응답 지연이 사용자 이탈로 직결되는 영역(코스 추천)에서는 LLM 호출을 의도적으로 배제하고,
> 사전 정제된 매핑 테이블과 결정론적 가중 점수만으로 작동합니다.
> 비용·지연·환각·재현 가능성 네 가지를 동시에 통제할 수 있습니다.
> LLM은 동행자 채팅·일지 생성처럼 진짜 필요한 곳에만 활용합니다."**

### 매번 다른 추천 — 3가지 이유

| 이유 | 설명 |
|---|---|
| 콘텐츠 빈약 인상 방지 | 매번 같은 결과 → 데이터 빈약해 보임 |
| 시연 안정성 | 발표·심사 시 청중이 매번 다른 화면 봄 |
| 재방문 유인 | 같은 사용자가 들어와도 새로운 코스 탐색 가능 |

→ `random.sample(score > 0 pool, 3)` 한 줄로 해결

### Q&A 권장 답

**Q. "AI가 어떻게 코스를 추천하나요?"**
> "사용자 입력(지역·기간·설화 카테고리 점수)으로 SQLite에서 후보 50개를 추출하고, 각 후보의 **설화 카테고리 점유율과 사용자 점수를 곱한 가중 점수**를 계산합니다. 점수가 양수인 풀에서 무작위 3개를 추출합니다. LLM 호출은 없습니다."

**Q. "왜 GPT를 코스 추천에 안 썼나요?"**
> "응답 지연이 사용자 이탈로 직결되는 영역에서는 LLM 호출을 의도적으로 배제했습니다. 코스 추천은 사전 정제된 매핑 테이블과 결정론적 가중 점수만으로 작동합니다. **비용·지연·환각·재현 가능성 네 가지를 동시에 통제**할 수 있고, AI는 동행자 채팅·일지 생성처럼 진짜 필요한 곳에 집중합니다."

**Q. "왜 매번 다른 추천이 나오나요?"**
> "점수 양수인 후보 풀에서 매번 `random.sample`로 3개를 뽑기 때문입니다. 카테고리 매칭이 전혀 안 되는 코스는 풀에서 자동 배제되므로 다양성을 유지하면서 추천 의미를 잃지 않습니다."

**Q. "1단계 점수 계산 자세히 설명해주세요."**
> "각 후보 코스의 장소들이 어떤 설화 카테고리와 매핑돼 있는지 점유율을 구하고, 사용자가 입력한 취향 점수와 점유율을 곱한 가중 합산입니다. 매핑된 지명의 구체성(specificity)이 높을수록 가중치가 커서 점수에 더 큰 영향을 줍니다. 예를 들어 '한라산'처럼 정확한 지명 매칭은 spec=10으로 가중치 2.0, '시 단위' 광역 매칭은 spec=5로 가중치 1.0입니다."

**Q. "지역을 '전체'로 선택하면 어떻게 동작하나요?"**
> "region 값이 '전체'면 SQL이 분기됩니다. 4지선다(북부/남부/동부/서부)는 요청 지역 + '전체' 태그 코스 297개 풀에서 추출하지만, '전체' 선택 시에는 region 제약을 풀고 1,090개 전체 코스에서 composite_score 상위 50개를 후보로 잡습니다. 그 다음 점수 계산·풀 필터·random.sample(3)는 4지선다와 동일합니다. 특정 지역에 얽매이지 않고 제주 전역에서 설화 카테고리 매칭이 가장 좋은 코스를 받고 싶은 사용자를 위한 옵션입니다."

## S4. 동행자 채팅 RAG + 페르소나

### 도착 감지 (CoreLocation 지오펜스)

| 모드 | 반경 | 체류 조건 |
|---|---|---|
| 도보 | 100m | 30초 |
| 차량 | 300m | 30초 |

30초 체류 = 잠깐 지나치는 오탐 방지. 시뮬레이터는 디버그 버튼.

### RAG 5단계 정확한 흐름 (`routers/travel.py:/companion`)

```
[입력] place_name, companion_type, message, history(8턴), folklore_summaries(선택)

[Retrieval]
  ① iOS가 보낸 folklore_summaries 우선 (클라이언트 캐시)
  ② 없으면 backend search_folklore_by_place():
      - place_name → text-embedding-3-small (1,536-d)
      - ChromaDB top-3 (코사인)
      - distance < 0.75 임계값 필터 ← 환각 억제
      - 포맷: "[설화 제목 / 주요장소] 본문 200자"

[Augmentation]
  CHARACTER_PROMPTS[companion_type]  ← 페르소나 시스템 프롬프트 + 퓨샷
  + "현재 장소: {place_name}\n이 장소의 설화:\n{retrieved}"
  + 최근 8턴 히스토리

  (설화 0건 fallback)
  "이 장소에는 별도 기록된 설화가 없습니다.
   설화가 없다는 사실 굳이 언급하지 말고,
   제주의 자연·바람·돌·바다·사람 이야기를
   당신의 캐릭터답게 자연스럽게 풀어내세요."

[Generation]
  ChatOpenAI(model="gpt-4o", temperature=0.7, streaming=True)
  async for chunk in llm_stream.astream(messages):
      yield SSE event { text: chunk }

[Client]
  iOS는 토큰 단위로 받아 화면 출력 + TTS 음성 동시 재생
```

### 환각 억제 2가지 메커니즘

| 메커니즘 | 내용 |
|---|---|
| 거리 임계값 | `distance < 0.75` 인 청크만 컨텍스트로 채택. 그 이상은 컷 |
| Fallback 프롬프트 | 설화 0건일 때 "제주 자연·바람·돌·바다 이야기로 페르소나답게" 지시 → 빈 응답 방지 |

### 페르소나 5종 ↔ 카테고리 매핑

| 페르소나 | 카테고리 | 말투 특징 |
|---|---|---|
| 마을 할망 👵 | 마을 공동체 전승 | 제주 사투리 11개 + 퓨샷 4개 ("혼저 옵서", "삼춘", "하영") |
| 당신·심방 🪬 | 무속신화·신격 전승 | 고어체 신탁 ("~이니라", "그대"), 단정적 |
| 영등신·해녀 🤿 | 해양·어촌 전승 | 해녀 어휘 6개 (테왁·숨비소리·불턱), 직선적 반말 |
| 도깨비 👺 | 생활민담·교훈담 | "크크크", 질문 되받기, 교훈 숨기기 |
| 도체비 👻 | 초자연 존재담 | "아마도", "아닌가?", 자기 부정, 문장 끊김 |

각 페르소나 = **독립 시스템 프롬프트 + 퓨샷 4개 + 공통 제약(3~4문장)**
같은 RAG 결과를 5종이 5가지 톤으로 풀어냄 → **콘텐츠 다양성의 핵심**

### 자동 배정 vs 수동 선택

| 위치 | 역할 |
|---|---|
| iOS (CompanionMapper 류) | 카테고리 점수 최고값 → 페르소나 자동 배정. CompanionPickerSheet로 수동 변경 가능 |
| 백엔드 `/travel/companion` | iOS가 보낸 companion_type을 VALID_CHARACTERS(5종) 검증만. 결정 로직 없음 |

### TTS — 5종 음성 동기화

- OpenAI TTS 5종 음성 (페르소나별 다름)
- 글자 출력 ↔ 음성 재생 거의 동시에 끝나도록 동기화
- 3개 재생 포인트:
  1. 장소 도착 오버레이 자동 인사
  2. 채팅 응답 (글자+음성 동시)
  3. 일지 화면 낭독 (페르소나 무관, 차분한 내레이터 톤)
- 우상단 스피커 1탭 = 전체 음소거 (영구 저장)

### Q&A 권장 답 — 핵심 4가지

**Q. "RAG 흐름 자세히 설명해주세요."**
> "장소 도착이 감지되면 iOS가 백엔드 `/travel/companion`을 호출합니다. 백엔드는 ① 장소명을 `text-embedding-3-small`로 1,536차원 벡터화, ② ChromaDB에서 코사인 유사도 top-3을 검색, ③ **distance 0.75 임계값**으로 저관련 청크를 컷, ④ 검색된 설화를 **페르소나 시스템 프롬프트와 최근 8턴 히스토리**에 주입, ⑤ GPT-4o로 SSE 토큰 스트리밍 응답합니다. iOS는 토큰을 받아 화면 출력과 페르소나 TTS를 동시에 재생합니다."

**Q. "환각은 어떻게 막았나요?"**
> "두 가지입니다. **코사인 distance 0.75 임계값**으로 저관련 청크를 컴퓨텍스트에서 컷해서 무관한 설화를 사실처럼 인용하는 것을 막고, 설화가 0건인 장소는 **fallback 프롬프트**로 '제주 자연·바람·바다 이야기를 페르소나답게 풀어달라'고 명시 지시해서 빈 응답이나 '모른다' 응답을 막습니다."

**Q. "왜 페르소나가 5종인가요?"**
> "사용자 설화 취향 카테고리가 다섯 가지인데, 단일 챗봇은 모든 톤을 흉내내려다 페르소나가 흐려집니다. 각 카테고리에 **독립 시스템 프롬프트와 퓨샷 4개**를 가진 페르소나를 매칭해, 같은 RAG 결과를 다섯 가지 말투로 풀어냅니다. 콘텐츠 다양성을 페르소나 분리로 만든 것입니다."

**Q. "도착 감지는 어떻게 하나요?"**
> "iOS CoreLocation 지오펜스로 도보 100m·차량 300m 반경 + 30초 체류 조건입니다. 30초는 잠깐 지나치는 오탐을 막기 위한 것입니다. 도착이 확정되면 동행자 오버레이가 전체화면으로 등장하면서 페르소나가 음성으로 자동 인사합니다."

## S5. 일지·민화·설화 가공

### 일지 + 민화 단일 호출 동시 생성 (`routers/travel.py:/journal`)

#### 입력 3가지 신호
| # | 신호 | 의미 |
|---|---|---|
| 1 | 방문 장소 시퀀스 | "비자림 → 성산일출봉 → 만장굴" |
| 2 | 동행자 대화 로그 | 장소당 최근 6턴, `--- {장소명} ---` 헤더로 묶음 |
| 3 | 커뮤니티 반응 통계 | `_get_community_context()`로 태그 집계 — "비자림: 47명 방문, '감동이에요(43%)'" 형식 |

→ 세 번째가 정량 통계 → 정성 일지로 변환되는 핵심.
같은 코스라도 페르소나·리뷰에 따라 매번 다른 일지.

#### 기술 — `with_structured_output(JournalLLMOutput)`
```python
class JournalLLMOutput(BaseModel):
    journal_text: str = Field(description="300~500자 한국어 1인칭 회고 일지")
    image_prompt: str = Field(description="민화 스타일 영어 이미지 프롬프트")

journal_llm = llm.with_structured_output(JournalLLMOutput)
```
Pydantic 스키마로 JSON 출력 강제 → 단일 호출에서 두 필드 동시 받음.
이미지 프롬프트는 `gpt-image-1`(1024x1024, quality medium)로 민화 생성.

#### 단일 호출 4가지 이유
| 이유 | 설명 |
|---|---|
| 비용 절감 | 2번 호출 → 1번 |
| 응답 지연 단축 | ~50초 → ~25-30초 |
| **의미적 일관성** | 텍스트와 이미지가 같은 장면 가리키도록 강제 |
| 부분 실패 견고성 | 이미지 생성 실패해도 일지 텍스트 정상 표시 |

### 설화 가공 3단계 — Progressive Disclosure (`services/folklore_llm.py`)

| 단계 | 보여주는 것 | 분량 | 모델 | temperature |
|---|---|---|---|---|
| **Lv1 후크** | 호기심 자극형 한 줄 | 30-50자 | gpt-4o-mini | 0.6 |
| **Lv2 장소-설화 연결** | 왜 이 장소에서 이 설화인가 | 30-60자 | gpt-4o-mini | 0.6 |
| **Lv3 풀스크린 스토리** | 5-7페이지 동화체 (프롤로그~에필로그 강제) | 페이지당 100-150자 | gpt-4o | 0.7 |

#### 점진적 깊이 디자인
- 사용자는 흥미가 동할 때만 다음 단계로 들어감
- 학술 원문 통째로 노출 대신, **호기심 → 맥락 → 깊이** 순서

#### 모델 분리 원칙
> **"짧은 한 줄은 저렴 모델, 길고 톤 일관성 필요한 건 비싼 모델."**

코스 추천에서 LLM을 빼는 의사결정과 같은 철학 — **자원을 진짜 필요한 곳에 집중**.

#### Lv3 풀스토리 추가 제약 (system prompt 강제)
- 첫 페이지 title = "프롤로그", 마지막 = "에필로그"
- 중간 페이지 = "1막", "2막", "3막" 순서
- 어투: 동화/구전 스토리텔링 ("옛날 옛적에", "~했답니다", "~였대요")
- 학술 요약 어투 금지

### 영구 캐시 — 같은 장소 두 번째 사용자는 LLM 호출 0회

| 테이블 | 저장 내용 |
|---|---|
| `folklore_connection_cache` | Lv2 장소-설화 연결 한 줄 |
| `folklore_story_cache` | Lv3 풀스토리 5-7페이지 |

→ 비용 절감 + 지연 0초 + 콘텐츠 일관성

### 민화 이미지 프롬프트 스타일 가이드 (LLM 강제 규칙)

```
반드시 다음으로 시작:

요소: 장소 (oreum hill, basaltic coast, traditional shrine, haenyeo diving)
      + 인물 (an old shaman grandmother, a haenyeo, a playful dokkaebi spirit)

```

### Q&A 권장 답 — 핵심 4가지

**Q. "일지는 어떤 데이터로 만드나요?"**
> "세 가지 신호입니다 — 방문 장소 시퀀스, 페르소나와 나눈 대화 로그, 그리고 다른 사용자들이 남긴 리뷰 태그 통계. 정량 통계를 정성 일지로 변환해, 같은 여행도 페르소나·리뷰에 따라 매번 다른 일지가 만들어집니다."

**Q. "왜 텍스트와 이미지를 한 번에 만드나요?"**
> "비용·지연 단축뿐 아니라 **텍스트와 이미지가 같은 장면을 가리키도록 의미적 일관성을 강제**하는 게 핵심 이유입니다. Pydantic의 `with_structured_output`으로 JSON 스키마를 강제해 한 번의 호출로 일지 텍스트와 이미지 프롬프트를 동시에 받습니다. 응답 시간이 50초에서 25~30초로 절반 단축되고, 이미지 생성이 실패해도 일지 텍스트는 정상 표시되어 부분 실패에 견고합니다."

**Q. "설화 가공 3단계는 무엇이고 왜 그렇게 했나요?"**
> "원본 설화는 학술 어투라 사용자가 보자마자 떠납니다. **점진적 깊이(progressive disclosure)** 디자인으로 3단계로 나눴습니다. Lv1은 한 줄 후크로 호기심을 자극, Lv2는 장소-설화 연결고리를 한 줄로, Lv3은 흥미를 느낀 사용자를 위한 5-7페이지 인스타 스토리 느낌의 풀스크린 동화체 뷰어입니다. **각 단계는 다른 모델을 사용**합니다 — 짧은 한 줄은 `gpt-4o-mini`, 길고 톤이 중요한 풀스토리만 `gpt-4o`로 자원을 분배했습니다."

**Q. "왜 영구 캐시를 두었나요?"**
> "장소-설화 조합은 시간이 흘러도 변하지 않는 콘텐츠입니다. 한 번 LLM이 생성한 결과를 SQLite에 영구 저장해두면, 그 다음 사용자는 LLM 호출 없이 즉시 응답을 받습니다. 비용 절감과 응답 지연 0초를 동시에 달성합니다."

## S6. 트레이드오프와 의사결정 (★ 부심교수 어필 핵심)

### ① 코스 → 설화 매핑 (vs 설화 → 코스 역설계)

| 시도 | 방식 | 결과 |
|---|---|---|
| 1차 | 설화 → 코스 역설계 (LangGraph 6노드) | 설화 GPS 분포가 희박해 동선 비현실. 폐기 |
| 2차 | 사용자 취향 점수 → 코스 필터 | 카테고리가 데이터 다양성 못 살림. 폐기 |
| **3차 (현재)** | **코스 → 설화 매핑** | 실제 데이터가 신뢰성·이동 효율 보장, AI는 큐레이션에 집중 |

핵심 사고:
- **여행자 입장에서도 "여행이 1순위, 설화가 그 위에 얹히는" 구조가 자연스러움**
- AI는 "동선 생성"이라는 어려운 일에서 물러나 "큐레이션·해석"이라는 잘하는 일에만 집중
- 9,134개 검증된 코스가 신뢰성 보장

### ② 페르소나 5종 분리 (vs 단일 챗봇)

| 단일 챗봇 + 톤 변수 | 5종 페르소나 완전 분리 (현재) |
|---|---|
| 시스템 프롬프트 1개 + 화자 변수 | **시스템 프롬프트 5개 독립** + 어휘·퓨샷 4개씩 |
| 같은 RAG → 비슷한 사고 흐름 | 같은 RAG → **다섯 가지 사고 흐름** |
| 톤만 표면적으로 바뀜 | 어휘·문장 구조·메타포 자체 다름 |
| 캐릭터성 흐림 | **콘텐츠 다양성 = 페르소나 차별** |

추가 이유 — UX 일관성:
- 사용자가 입력 단계에서 5종 카테고리를 선택
- **카테고리 ↔ 페르소나 1:1 매핑**으로 입력과 결과 사이의 일관성

### ③ 영구 캐시 (vs 매번 LLM)

> **"같은 콘텐츠는 한 번만 생성, 영원히 재사용."**

| 가치 | 효과 |
|---|---|
| 비용 절감 | 두 번째 사용자는 LLM 호출 0회 |
| 응답 지연 | 0초 |
| 콘텐츠 일관성 | 같은 장소·설화 조합 = 항상 같은 결과 |

### ④ 코스 추천에 LLM 미사용 (vs LLM 활용) — 가장 강한 트레이드오프

핵심 표현 (외울 것):
> **"LLM은 비용·지연·환각·재현 가능성 네 가지 트레이드오프를 동반합니다.
> 사전 정제된 데이터와 결정론적 알고리즘으로 해결 가능한 영역에서는 그 네 가지를 모두 통제할 수 있고,
> 코스 추천이 정확히 그 영역에 해당합니다.
> 반대로 동행자 채팅·일지 생성처럼 자연어 창의성이 핵심인 영역에는 LLM이 필수입니다.
> **자원을 진짜 필요한 곳에 집중하는 분배 원칙**이 시스템 전반의 설계 철학입니다."**

같은 철학의 다른 적용:
- 설화 가공 3단계도 **Lv1·Lv2는 gpt-4o-mini, Lv3만 gpt-4o** — 모델 분리
- 일지 + 민화도 단일 호출로 묶음 — 호출 횟수 최소화

### ⑤ Firebase 도입 (vs 익명 device_id만)

| 이유 | 설명 |
|---|---|
| ① 다기기 동기화 | 같은 사용자가 iPhone + iPad — 익명 ID는 분리, Firestore는 user_id 기준 통합 |
| ② 데이터 영속성 | 디바이스 변경·재설치 시 익명 ID 새로 발급, Firebase uid는 유지 → 일지·코스 보존 |
| ③ 출시 로드맵 | App Store 출시 + 실사용자 확보가 졸업 발표 이후. 표준 인증 인프라 필수 |

추가: 백엔드 모든 사용자 식별 엔드포인트는 Firebase ID Token 검증으로 보안 확보.

### Q&A 권장 답 — 핵심 5가지

**Q. "왜 설화에서 코스를 만들지 않고 반대로 만들었나요?"**
> "처음엔 설화에서 출발해 코스를 만드는 역설계 방식을 시도했는데, 설화 GPS 분포가 희박해 동선이 비현실적이었습니다. 그래서 반대로 9,134개 검증된 실제 코스를 베이스라인으로 깔고, 그 위에 AI가 설화를 매핑하는 구조로 바꿨습니다. AI는 '동선 생성'이라는 어려운 일에서 물러나 '큐레이션·해석'이라는 잘하는 일에 집중합니다."

**Q. "왜 페르소나가 5종인가요? 단일 챗봇이 더 단순하지 않나요?"**
> "단일 챗봇은 톤만 표면적으로 바뀝니다. 5종 페르소나는 시스템 프롬프트와 어휘·퓨샷이 모두 독립이라 같은 RAG 결과도 다섯 가지 사고 흐름으로 풀어냅니다. **콘텐츠 다양성을 페르소나 분리로 만들었습니다**. 또 사용자가 입력 단계에서 5종 카테고리를 선택하므로 카테고리-페르소나 1:1 매핑으로 UX 일관성도 확보됩니다."

**Q. "왜 GPT를 코스 추천에 안 썼나요?"**
> "LLM은 비용·지연·환각·재현 가능성 네 가지 트레이드오프를 동반합니다. 사전 정제된 데이터와 결정론적 알고리즘으로 해결 가능한 영역에서는 그 네 가지를 모두 통제할 수 있고, 코스 추천이 정확히 그 영역입니다. 반대로 동행자 채팅·일지 생성처럼 자연어 창의성이 핵심인 영역에는 LLM이 필수입니다. **자원을 진짜 필요한 곳에 집중하는 분배 원칙**이 시스템 전반의 설계 철학입니다."

**Q. "영구 캐시의 핵심 가치는?"**
> "같은 콘텐츠는 한 번만 생성하고 영원히 재사용합니다. 비용·지연·일관성 세 가지를 한 번에 잡습니다."

**Q. "왜 익명이 아닌 Firebase를 도입했나요?"**
> "세 가지 이유입니다. 다기기 동기화, 디바이스 변경 시 데이터 영속성, 그리고 졸업 이후 App Store 출시·실사용자 확보 로드맵에서 표준 인증 인프라가 필수이기 때문입니다."

점수 변환 (iOS):
```
사용자: 메인 1개 + 사이드 1개 선택 (총 2개)
  ↓
category_scores = {
  메인:   3,
  사이드: 1,
  나머지: 0,
}
  ↓
POST /course/list
```

변환 이력:
- 초기: 이야기 결 4지선다 + 배경 4지선다 → 16조합 변환표 → 5개 점수 (복잡)
- 현재: 5개 카테고리 직접 선택 + 메인 3점 / 사이드 1점 (단순화)

### A.3 비짓제주 정제 + 사용자 입력 → 추천 (런타임 흐름)

```
[1] iOS 입력: region (5지선다: 북부/남부/동부/서부/전체), 메인 카테고리, 사이드 카테고리, duration_days
[2] iOS 변환: category_scores = {메인:3, 사이드:1, 기타:0}
[3] POST /course/list
[4] course_list_agent.run_course_list():
    ├─ curated_courses LIMIT 50
    │   · region=="전체" → 제약 없음 (1,090개 모두 후보)
    │   · 그 외 4지선다 → region 일치 우선 + '전체' 태그 코스 보완 (~297개)
    │   · 공통: place_count >= max(3, duration_days)
    ├─ 각 후보 _score_course() — specificity 가중 + 카테고리 점유율 × 사용자 점수
    ├─ score > 0인 후보 풀
    └─ random.sample(pool, 3)
[5] 3개 코스 (각 코스의 장소 목록 GPS 포함)
[6] 사용자 선택 → POST /course/detail
[7] course_detail_agent.run_detail_agent():
    ├─ course_places로 장소 목록 조회
    ├─ 각 장소: place_folklore_mapping LIMIT 20
    ├─ 카테고리 점수 + specificity 재정렬
    └─ 장소당 folklore_pins[:3] 첨부
[8] iOS 미리보기 화면 (지도 + 장소 카드 + Lv1 후크)
```

**전체 흐름 LLM 호출 0회** — SQLite + 결정론적 함수만.

### A.4 RAG 사용처 정확히 1곳

| 위치 | 임계값 | 왜 이 임계값? |
|---|---|---|
| `/travel/companion` 동행자 채팅 | **distance < 0.75** | 장소명 쿼리는 컨텍스트 명확. 약간 느슨해도 페르소나가 자연스럽게 풀어냄 |

**RAG가 사용 안 되는 곳**:
- 코스 추천 1·2단계 — 결정론
- 설화 Lv1/2/3 가공 — 원본 직접 LLM 전달
- 일지·민화 생성 — 방문 이력 + 대화 + 리뷰만

→ **RAG는 "사용자가 자연어로 묻는 순간"에만 활성**. 정적 콘텐츠 생성은 모두 정형 데이터 기반.

## 부록 B. 데이터 가공 실사례 — 「각시당본풀이」 (B1 모듈)

### 「각시당본풀이」(C_M_001) 한 건이 4단계를 거치는 모습

#### 1️⃣ 원본 수집 (`fetch_metadata.py`)
제주특별자치도 API로부터 받는 메타데이터 + 원문 HTML
- code_no: `C_M_001`
- title: `각시당본풀이`
- category: 본풀이
- 원문: HTML 태그 포함된 텍스트

#### 2️⃣ 정제 (`extract_text.py` + `normalize_text.py`)
HTML 태그 제거 + 공백 정리
- `clean_text`: 1,164자
- `normalized_text`: 1,162자
- 첫 문장: "C_M_001 각시당본풀이 / 1 개요 / 각시당본풀이는 제주시 삼도동에 있던 각시당에 얽힌 무속신화이다..."

#### 3️⃣ 청크 분할 (`build_chunks.py`, 900자 + 120 overlap)

```
1,162자 → 2개 청크
[청크 0] start=0,   end=900   → 길이 900자
[청크 1] start=780, end=1162  → 길이 382자
         ↑ 900 - 120(overlap) = 780부터 시작

오버랩 시각화
0─────────────────────────────900──────────1162
[──────── 청크 0 (900자) ────────]
                          [── 청크 1 (382자) ──]
                          ↑
                          780 (overlap 120자 공유)
```

#### 4️⃣ 임베딩 (`ingest_to_vector_db.py`)
- 각 청크 → OpenAI `text-embedding-3-small` → 1,536차원 벡터
- ChromaDB `jeju_folklore_chunks` 컬렉션 저장
- 메타데이터: doc_id, source_type, code_no, title, category, chunk_index
- 검증: ChromaDB 직접 조회 결과 C_M_001 청크 2개 모두 임베딩 완료 ✅

### 발표용 멘트 (활용)
> "예를 들어 「각시당본풀이」라는 설화는 1,162자 본문이 900자 + 120자 overlap의 슬라이딩 윈도우로 2개 청크로 분할되고, 각각 1,536차원 벡터로 변환된 뒤 ChromaDB에 저장됩니다. 505건 전체로는 약 1,749개 청크가 됩니다."

## 부록 C. 코사인 유사도·임계값·차원 (Q&A 대비)

### 코사인 유사도
- 두 벡터의 방향 비교 (길이 무시)
- 수식: `cosine_similarity = (a·b) / (‖a‖×‖b‖)`
- ChromaDB distance: `cosine_distance = 1 - cosine_similarity`
- 같은 방향 → distance 0, 정반대 → distance 2
- 텍스트 임베딩 표준 (유클리드 거리 안 씀 — 벡터 길이는 텍스트 분량에 따라 변동, 의미는 방향에 담김)

### distance 0.75 임계값
- 시행착오 튜닝 값
- 관련성 vs 환각 트레이드오프 균형점
  - 너무 낮으면 (0.3-0.5): 거의 일치만 통과 → 빈 응답
  - 너무 높으면 (1.0+): 무관 청크 통과 → 환각 ↑
- **0.75는 "관련성을 양보하더라도 빈 응답을 피하는" 선**

### 1,536차원
- OpenAI `text-embedding-3-small` 모델의 고정 스펙
- 본인이 정한 값 아님 (모델 선택의 결과)
- 모델 선택 근거:
  - vs `3-large` (3,072차원): 6배 싸고 한국어 검색에 충분
  - vs `ada-002` (1,536차원): 5배 싸고 성능 더 높음

### 발표용 간결 멘트
- distance 0.75: "관련 청크는 통과시키고 무관 청크는 컷하기 위해 0.75로 설정"
- 차원 1,536: "OpenAI text-embedding-3-small 모델 스펙"
- 코사인 유사도: 발표에선 생략, Q&A 들어오면 "두 벡터 방향이 비슷한지 측정"
