연구 배경

제주는 연 1,500만 명이 찾는 한국 최대 관광지이자, 중요 무형 문화유산의 보고이다.

문제: 설화·민담 데이터는 공공 API로 개방되어 있지만, 실제 관광 경험과 결합된 형태로 소비되지 못한다.
텍스트 그대로는 길고 어려우며, 어떤 장소에서 어떤 이야기를 들어야 할지 알 수 없다.

최근 LLM · RAG · AI 에이전트 기술의 발전으로, 정적인 텍스트 데이터를 현장 맥락에 맞게 가공·전달하는 것이 비로소 현실적으로 가능해졌다. [Lewis 2020] [Yao 2023]

04 / 23 02 · Problem & Value

문제 정의 & 핵심 가치

"실제 검증된 여행 코스 위에 설화를 얹는다."
기존 접근의 한계
· 설화 기반 코스를 처음부터 LLM이 만들면 지명·동선이 비현실적
· 설화 본문은 길고 어려운 어휘 → 일반 관광객 접근성 낮음
· 위치 기반 콘텐츠는 있으나 무형 문화유산과 결합 부재
탐라담의 해법
· Visit Jeju 9,134개 실제 코스 위에 설화를 매핑
· 코스 현실성은 실제 데이터가 보장
· 설화는 현장에서 AI 동행자가 이야기로 들려줌
· 여행 후 GPT-4o가 여행 일지 + 민화 이미지 생성
4-Step Value Loop:   검증된 코스 추천  →  설화 매핑  →  현장 AI 동행자  →  일지·민화 기록

08 / 23 06 · Design Pivots

설계 변경 이력 — 3차 재설계

차수접근한계 또는 채택 사유
1차 설화 → 코스 역설계
설화 N개 선택 → LLM이 동선 생성		 LLM이 생성한 좌표·동선이 비현실적. 같은 장소가 떨어진 일정에 반복 등장. 실제 가용 시간 미고려.
2차 설화 취향 점수 기반
취향 벡터 ↔ 코스 벡터 유사도		 점수 분포가 평탄해 변별력 부족. "왜 이 코스인지" 사용자에게 설명 불가능. 매번 동일 결과 → 재방문 동기 결핍.
3차 (현재) 코스 → 설화 매핑
검증 코스 위에 설화 레이어		 채택. 코스 현실성은 Visit Jeju 데이터가 보장, 설화는 컨텍스트 레이어로 분리. 추천 다양성·설명력·확장성 모두 확보.
교훈: LLM에게 모든 것을 맡기지 않는다. 검증된 데이터가 보장할 수 있는 영역과 LLM이 가치를 더하는 영역을 분리하는 것이 핵심이었다.

09 / 23 07 · Course Recommendation

코스 추천 — 2단계 API 파이프라인

① POST /course/list — 후보 3개 선별
· LangGraph course_list_agent
· 취향·지역·일수 필터링 → 후보 풀에서 30개 무작위 추출
· GPT-4o가 30개를 읽고 3개를 선별 + 추천 이유 1줄
· 매번 다른 결과 (시연 다양성·재방문 동기 확보)
· 평균 응답 ~8초
② POST /course/detail — 상세 + 내러티브
· 결정론적 함수 (선택된 코스 ID에 종속)
· Haversine 거리 계산 → 코스 장소 반경 3km 내 설화 매핑
· GPT-4o가 코스 내러티브 2문장 생성 (LLM 호출 1회)
· 같은 코스는 같은 결과 → SQLite 캐시 적중률 高
· 비용 안정 · 평균 응답 ~3초

10 / 23 08 · Home Screen

홈 화면 — 첫 진입 임팩트

"검증된 추천 + 설화 문화 큐레이션" 임을 홈 화면 한 번에 전달한다.
① 오늘의 설화 카드
· 날짜를 시드로 사용한 결정론적 셔플 — 같은 날 같은 설화
· 호기심 후크 1줄 + 한 문장 요약
· 탭 시 풀스크린 스토리 뷰어로 진입
② 제주 지도 + 설화 핀
· 지도 위에 설화 위치 핀 표시
· "이 섬은 이야기로 덮여 있다"는 시각적 메시지
· 핀 탭 → 해당 설화 상세
③ 추천 코스 3선
· /course/list 결과 카드
· 코스 이름 + AI 추천 이유 1줄
· CTA → 상세 미리보기 → 탐험 시작
설계 의도: 가입·튜토리얼 없이 첫 화면에서 가치를 즉시 체감하도록 했다. (앱스토어 평가 화면 1초의 법칙)

11 / 23 09 · Place Detail

장소 상세 — 설화 데이터 3단계 가공

원본 설화는 길고 어려운 어휘 + 코드(◆, [1] 등) 혼재. 3단계 점진적 노출로 인지 부담을 분산한다.
레벨노출 형태처리 방식예시
Lv1 장소 카드 부제 (호기심 후크 1줄) 원문 전처리: 코드 제거 + 1줄 후크 추출 (LLM 1회 → 캐시) "심방으로 살다 다시 이승으로 돌아온 굴묵밭 할망"
Lv2 장소-설화 연결 1줄 장소 ↔ 설화 관계 요약 LLM 호출 → folklore_connection_cache 영구 저장 "이 마을의 본향당 신을 모시는 이야기"
Lv3 풀스크린 스토리 뷰어 5~7페이지 동화체 리라이팅 → folklore_story_cache 영구 저장. 인스타 스토리 UX (좌우 스와이프, 자동 진행) 아동도 읽을 수 있는 동화 문체로 변환된 본문 + 페이지별 표지
비용 최적화: Lv1·Lv2·Lv3 결과는 모두 백엔드 영구 캐시. 동일 장소·설화 조합은 LLM 0회 호출. → 출시 후 사용자 1만 명에서도 LLM 비용 선형 증가하지 않음.

13 / 23 11 · Explore Mode

GPS 탐험 모드 & 도착 감지

기술 스택
· iOS CoreLocation — 백그라운드 위치 권한
· 지오펜스(Geofence) — 코스 내 장소마다 자동 등록/해제
· 배터리 보존을 위해 정확도 동적 조정 (이동 중 ↑, 정지 시 ↓)
도착 판정 로직
· 도보 100m / 차량 300m 진입
· 동시에 30초 이상 체류 (오탐 방지)
· 두 조건 동시 충족 시 도착 이벤트 발화
· 시뮬레이터에서는 도착 시뮬레이션 버튼으로 즉시 트리거
도착 이벤트 발화 즉시 AI 동행자 오버레이가 전체화면으로 등장. 사용자는 별도 조작 없이 "도착했다 → 이야기가 시작된다"는 마법 같은 순간을 경험한다.
트레이드오프 — "차량 이동 중 오탐 방지" 설계 이유
이동 경로상 코스 외 장소를 잠시 스치고 지나갈 때 오버레이가 뜨면 운전 위험 + UX 망가짐. 30초 체류 + 반경 조건으로 실제 "방문"한 경우에만 발화하도록 했다.

5종 태그
소름 😨 감동 🥹 신기 🤔 무서워요 😱 역사적 🏛

다중 선택 가능
한 줄 감상
최대 200자 자유 입력. 입력 부담 최소화 — 한 줄이라도 남기는 것이 목표.

· 입력: 방문 장소 + 설화 + 사용자 태그·감상
· 출력: 1인칭 회고 300~500자 (구조화 JSON)
· 단일 호출로 일지 + 이미지 프롬프트 동시 생성
· 사용자의 감상 단어가 자연스럽게 일지에 녹아듦
🎨 민화 이미지 — gpt-image-1
· 한국 전통 민화 스타일 강제 프롬프트
· 평면 색감 · 두꺼운 검은 외곽선 · 종이 질감
· "AI 슬롭"이 아닌 고유한 시각 아이덴티티
· 해당 코스의 핵심 장소·설화 모티프 반영
성능 · 사용자 경험
· 응답 시간 ~25~30초 → "이야기 정리 중…" 로딩 화면으로 기대감 연출
· 단일 호출로 일관성 확보 (일지와 이미지가 동일 코스 이해)
공유 (SNS Loop)
· iOS ShareLink 네이티브 공유 시트

· 자연스러운 바이럴 채널 (SNS 쇼츠 마케팅 핵심)

STEP 06

하나의 앱에서 여행 전(코스 추천) · 여행 중(AI 동행자) · 여행 후(일지·민화·공유)를 단일 흐름으로 묶음.
③ 안정적 코스 생성
Visit Jeju 9,134개 실제 코스를 기반으로 함 → LLM 환각으로 인한 비현실적 동선·잘못된 좌표 문제를 구조적으로 차단.
