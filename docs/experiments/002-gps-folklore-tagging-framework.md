# 실험 002. GPS 기반 활용 가능 설화 태깅 프레임워크

**날짜**: 2026-04-08  
**목적**: 실제 코스 추천에 활용 가능한 설화만을 기준으로 내부 카테고리를 도출하고, 이후 사용자 질문을 역설계하기 위한 태깅 기준을 수립합니다.

---

## 1. 왜 이 작업이 필요한가

기존 구조는 사용자의 입력을 먼저 정해진 5개 내부 테마로 매핑한 뒤, 해당 테마를 ChromaDB 검색 쿼리로 확장하는 방식이었습니다.

하지만 현재는 다음 문제가 확인되었습니다.

- 5개 테마가 실제 설화 데이터에서 자연스럽게 도출된 분류라고 보기 어렵습니다.
- 일부 사용자 입력은 결과가 지나치게 단조롭게 매핑됩니다.
- 질문을 먼저 정한 뒤 설화를 맞추는 방식보다, 설화를 먼저 분석한 뒤 질문을 역설계하는 편이 더 자연스럽습니다.

따라서 이번 단계에서는 먼저 **GPS가 있어 실제 코스 추천에 활용 가능한 설화 228개**를 분석하고, 그 결과로 내부 카테고리를 도출합니다.

---

## 2. 분석 대상

### 대상 데이터
- 파일: `data/processed/folklore_gps.json`
- 총 개수: **228개**
- 구성:
  - `legend`: 142개
  - `folktale`: 86개

### 왜 228개를 먼저 보는가
- 전체 505개 설화 중 실제 코스 추천에 바로 활용 가능한 설화만 선별된 집합입니다.
- 내부 카테고리는 서비스에 실제로 쓰이는 데이터 기준으로 만들어야 합니다.
- 이후 에이전트 설계, 질문 설계, 코스 품질 개선과 직접 연결됩니다.

---

## 3. 태깅 원칙

### 3-1. 단일 라벨이 아니라 멀티태그 방식으로 태깅합니다

제주 설화는 한 이야기 안에 여러 성격이 동시에 섞여 있는 경우가 많습니다.

예:
- 무속신화 + 마을 전승
- 해양 전승 + 신격 이야기
- 지명 유래 + 초자연 존재담

따라서 설화 1개당 카테고리 1개를 강제로 붙이는 대신, 여러 태그를 함께 기록합니다.

### 3-2. 학술 분류보다 서비스 활용성을 우선합니다

좋은 태그는 다음 조건을 만족해야 합니다.

- 사용자 질문과 연결될 수 있어야 합니다.
- 설화 검색 및 후보 추출에 도움이 되어야 합니다.
- 지역, 장소, 여행 코스 설계와 연결되어야 합니다.
- 에이전트가 선택과 조합을 할 때 의미 있는 기준이어야 합니다.

### 3-3. 내부 카테고리는 태깅 결과에서 나중에 도출합니다

처음부터 "최종 카테고리"를 확정하지 않습니다.  
먼저 설화별 태깅을 진행하고, 반복되는 패턴을 묶어 내부 카테고리를 정의합니다.

---

## 4. 태깅 축

각 설화는 아래 6개 축으로 기록합니다.

### A. 서사 유형
- `무속신화`
- `전설`
- `민담`
- `혼합형`

기준:
- 본풀이, 당신, 심방, 굿, 좌정 중심이면 `무속신화`
- 특정 장소/인물/유래 중심이면 `전설`
- 생활 교훈, 해학, 인간사 중심이면 `민담`

### B. 핵심 내용
- `신격/본풀이`
- `지명·지형 유래`
- `해양·어촌`
- `초자연 존재`
- `인간사 비극`
- `가족·인연`
- `생활·해학`
- `교훈·권선징악`
- `마을 전승`
- `자연물·산천`

이 축이 나중에 내부 카테고리 도출의 핵심 기준이 됩니다.

### C. 주요 존재
- `신`
- `인간`
- `조상신`
- `마을신`
- `도체비/요괴`
- `혼령/귀신`
- `동물`
- `용왕/해신`
- `심방`
- `왕/관원/선비`

### D. 배경 공간
- `바다`
- `해안 마을`
- `산/오름`
- `동굴/연못/폭포`
- `마을`
- `당/사당`
- `섬`
- `들판/길`
- `한라산 권역`

### E. 정서/분위기
- `신성`
- `신비`
- `공포`
- `슬픔`
- `따뜻함`
- `비장함`
- `익살`
- `경외`

이 축은 나중에 사용자 질문과 직접 연결됩니다.

### F. 서사 기능
- `기원 설명`
- `금기 설명`
- `장소 신성화`
- `인물 전승`
- `교훈 전달`
- `공동체 기억`
- `비극 기억`

## 5. 기록 형식

설화 1개를 태깅할 때 아래 항목을 기록합니다.

### 기본 메타데이터
- `code_no`
- `title`
- `source_type`
- `primary_place`
- `lat`
- `lng`

### 태깅 항목
- `narrative_type`
- `core_topics`
- `entities`
- `spaces`
- `moods`
- `functions`

### 보조 메모
- `summary_hint`: 기존 요약 또는 핵심 메모
- `tag_notes`: 왜 그렇게 태깅했는지 간단한 근거

---

## 6. 내부 카테고리 도출 방식

태깅이 충분히 쌓이면, 아래와 같은 방식으로 내부 카테고리 초안을 도출합니다.

### 예시 후보
- `무속신화·신격 전승`
- `지명·지형 유래`
- `해양·어촌 전승`
- `초자연 존재담`
- `인간사 비극·가족담`
- `생활민담·교훈담`
- `마을 공동체 전승`

주의할 점:
- 이 목록은 확정안이 아닙니다.
- 실제 태깅 결과를 본 뒤 합치거나 나눌 수 있습니다.
- 최종 카테고리는 사용자 질문과 에이전트 설계에 모두 도움이 되는 수준으로 정리해야 합니다.

---

## 7. 질문 역설계 원칙

질문은 카테고리 도출이 끝난 뒤 역설계합니다.

즉 순서는 다음과 같습니다.

```text
설화 태깅
→ 반복되는 내용 축 확인
→ 내부 카테고리 정의
→ 카테고리를 잘 구분할 수 있는 질문 설계
```

현재 질문이 `분위기 / 장소 / 일수 / 이동수단`이라면, 이후에는 설화 분석 결과에 따라 다음과 같은 질문으로 발전할 수 있습니다.

- 신성한 장소의 기원 이야기가 좋은가, 사람 사는 이야기가 좋은가
- 초자연적이고 으스스한 분위기가 좋은가, 따뜻한 인간사가 좋은가
- 바다와 어촌 전승이 좋은가, 산과 지형 유래가 좋은가
- 마을 공동체 전승이 좋은가, 특정 인물이나 신격 전승이 좋은가

즉 질문은 UX에서 먼저 정하는 것이 아니라, **설화 분류 체계를 가장 잘 반영하도록 나중에 설계**해야 합니다.

---

## 8. 진행 순서

1. GPS가 있는 228개 설화 목록을 작업 시트로 정리합니다.
2. 30개 정도를 먼저 태깅해 태그 체계가 과하거나 부족하지 않은지 확인합니다.
3. 70개 수준까지 늘려 반복되는 패턴을 파악합니다.
4. 내부 카테고리 v1을 도출합니다.
5. 228개 전체에 대해 태깅을 완료합니다.
6. 내부 카테고리를 기준으로 사용자 질문 세트를 역설계합니다.
7. 최종적으로 에이전트의 후보 선택 구조와 연결합니다.
