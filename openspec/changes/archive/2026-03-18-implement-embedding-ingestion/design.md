## Context

현재 프로젝트는 `scripts/build_chunks.py`를 통해 제주 설화·민담 텍스트를 SQLite 데이터베이스(`storage/metadata.db`)의 `chunks` 테이블에 저장한 상태입니다. 다음 단계는 이 텍스트들을 검색 가능한 벡터 형태로 변환하여 벡터 데이터베이스에 저장하는 것입니다.

## Goals / Non-Goals

**Goals:**
- SQLite `chunks` 테이블의 데이터를 읽어 벡터 임베딩 생성.
- 생성된 임베딩과 메타데이터(제목, 출처 등)를 벡터 DB(ChromaDB)에 저장.
- 이미 처리된 청크를 건너뛰는 효율적인 동기화 로직 구현.
- 검색 성능 향상을 위한 메타데이터 필터링 구조 설계.

**Non-Goals:**
- 실제 사용자 검색 인터페이스 구현 (이는 별도의 변경 사항으로 다룸).
- 고도화된 하이브리드 검색(BM25 + Vector) 구현.

## Decisions

- **임베딩 모델**: OpenAI의 `text-embedding-3-small`을 사용합니다. 한국어 의미 파악 능력이 뛰어나고 비용 효율적입니다.
- **벡터 데이터베이스**: `ChromaDB`를 사용합니다. 별도의 서버 설치 없이 로컬 파일(`storage/vector_db/`)로 관리할 수 있어 현재 프로젝트 구조에 적합합니다.
- **메타데이터 구조**: 벡터 DB에는 `doc_id`, `source_type`, `code_no`, `title`, `category`, `tags`를 함께 저장하여 검색 시 필터링이 가능하도록 합니다.
- **배치 처리**: API 호출 효율을 위해 청크를 일정한 크기(예: 100개)로 묶어 임베딩을 요청합니다.

## Risks / Trade-offs

- **비용**: OpenAI API 사용에 따른 비용이 발생합니다. (단, 현재 데이터 규모상 매우 소액일 것으로 예상됨)
- **속도**: 네트워크 API를 호출하므로 수천 개의 청크 처리 시 시간이 소요될 수 있습니다. 이를 위해 진행 상태를 출력하도록 합니다.
- **의존성**: `chromadb`, `openai` 라이브러리가 추가로 필요합니다.
