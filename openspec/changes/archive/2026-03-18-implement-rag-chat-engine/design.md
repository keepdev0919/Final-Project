## Context

임베딩과 벡터 데이터베이스 적재(`implement-embedding-ingestion`)가 완료된 시점을 전제로 합니다. 이제 사용자가 던지는 질문(Query)을 처리하고, ChromaDB에서 가장 관련성 높은 설화 조각들을 찾아낸 뒤, OpenAI의 LLM을 통해 최종 답변을 생성하는 과정을 설계합니다.

## Goals / Non-Goals

**Goals:**
- 사용자 질문을 벡터로 변환하여 ChromaDB에서 유사도 검색(Vector Search).
- 검색된 결과(Context)를 시스템 프롬프트와 결합하여 LLM 답변 생성.
- 답변의 근거가 된 설화의 제목과 출처 정보를 명확히 제시.
- 터미널 기반의 인터랙티브 대화 루프 구현.

**Non-Goals:**
- 웹 UI 또는 모바일 앱 인터페이스 구축 (이는 추후 별도 단계).
- 사용자 세션 관리 및 대화 이력의 영구 저장 (메모리 내 저장으로 제한).

## Decisions

- **검색 방식**: ChromaDB의 `similarity` 검색을 기본으로 사용하며, 필요시 `n_results`를 조절하여 컨텍스트 창 크기에 맞춥니다.
- **환각 억제 방식**: 검색 결과 중 거리(`distance`)가 임계값보다 큰 청크는 버려 낮은 관련성 문맥이 답변 생성에 섞이지 않도록 합니다.
- **LLM 모델**: 기본값은 `gpt-4o`로 두고, 필요시 CLI 인자로 다른 OpenAI 채팅 모델을 지정할 수 있게 합니다.
- **질문 임베딩 모델**: 문서 적재와 동일하게 `text-embedding-3-small`을 사용하여 질의-문서 벡터 공간을 일치시킵니다.
- **프롬프트 전략**: "당신은 제주 설화 전문가입니다. 제공된 설화 텍스트만을 근거로 답변하되, 모르는 경우 솔직하게 답변하십시오"라는 지침을 포함합니다.
- **출처 표기**: 모델 자유 생성에 맡기지 않고, 답변 생성 후 애플리케이션 레이어에서 "출처: [제목] (코드: [번호])" 형식으로 고정 부착합니다.
- **대화 인터페이스**: 별도 웹 UI 없이 `scripts/chat_engine.py`에서 CLI 루프를 제공합니다.
- **메모리 전략**: 최근 3턴 정도의 Q/A만 프롬프트에 포함하는 경량 메모리 전략을 사용합니다.

## Risks / Trade-offs

- **환각(Hallucination)**: LLM이 제공되지 않은 정보를 지어낼 위험이 있습니다. 이를 방지하기 위해 프롬프트 제약을 강화합니다.
- **토큰 비용**: 검색된 컨텍스트가 길어질수록 모델 호출 비용이 증가합니다.
- **방언 이해도**: 표준어 정규화 과정을 거쳤으나, 여전히 고유 명사나 특수한 명칭에 대한 오독 가능성이 있습니다.
