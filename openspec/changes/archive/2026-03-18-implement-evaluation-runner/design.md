## Context

현재 챗 엔진은 단일 질의 또는 CLI 대화형 질의는 처리할 수 있지만, 고정 질문셋 20개를 자동으로 돌려 결과를 남기는 기능은 없다. 평가 기록은 졸업프로젝트 보고서와 반복 비교에 직접 쓰이므로, 사람이 읽기 쉬운 Markdown 결과와 후속 자동 분석에 쓸 JSONL 결과를 함께 남기는 구조가 필요하다.

## Goals / Non-Goals

**Goals:**
- 질문셋 JSONL을 순차 실행하는 평가 실행기 제공
- 질문별 검색 결과와 최종 답변, 간단한 통과 여부를 파일로 저장
- `chat_engine.py`의 검색/생성 로직을 재사용해 평가와 실제 챗 엔진 간 동작 차이를 줄임

**Non-Goals:**
- 정교한 LLM 기반 자동 채점
- 웹 대시보드 제공
- 장기 이력 저장이나 DB 기반 평가 관리

## Decisions

- **입력 형식**: 기존 [evaluation_questions.jsonl](/Users/choikjun/Desktop/keepdev/졸프/data/processed/evaluation_questions.jsonl)을 그대로 사용한다. 별도 포맷을 만들지 않아 질문 관리가 한 군데로 유지된다.
- **실행 방식**: 신규 `scripts/evaluation_runner.py`가 질문셋을 순회하며 챗 엔진 공통 함수를 호출한다. 별도 서브프로세스로 `chat_engine.py`를 반복 실행하지 않고 파이썬 함수 호출로 묶어 비용과 중복을 줄인다.
- **출력 형식**: 기계 처리용 `jsonl`과 사람이 읽는 `md`를 동시에 생성한다. JSONL은 질문별 상세 결과, Markdown은 요약표와 핵심 로그 중심으로 구성한다.
- **간단 평가 로직**: 초기 버전은 규칙 기반으로만 판정한다. `rejection_expected=true`인 질문은 거부 문구와 `출처: 없음`을 확인하고, 일반 질문은 필수 키워드 포함 여부와 출처 존재 여부를 확인한다.
- **파라미터 노출**: `--limit`, `--source`, `--k`, `--max-distance`, `--chat-model`, `--embedding-model`을 CLI 인자로 노출해 같은 질문셋으로 설정별 비교가 가능하게 한다.

## Risks / Trade-offs

- **규칙 기반 판정 한계** → 키워드 기반 평가는 의미적으로 맞는 답변도 일부 놓칠 수 있다. 초기 자동 평가는 참고 지표로만 사용하고, 최종 품질 평가는 수동 검토와 병행한다.
- **API 비용 증가** → 질문셋 전체 실행은 OpenAI 호출을 반복하므로 비용이 발생한다. `--limit` 옵션과 소규모 질문셋 샘플 실행으로 비용을 조절한다.
- **코드 중복 위험** → 챗 엔진과 평가 실행기가 로직을 따로 가지면 결과가 어긋날 수 있다. 공통 함수 재사용으로 완화한다.
