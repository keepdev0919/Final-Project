# 제주 설화·민담 챗봇 전체 파이프라인

## 개요

이 프로젝트의 목표는 제주 설화와 민담 데이터를 수집하고, 이를 검색 가능한 벡터 DB로 구성한 뒤, 최종적으로 사용자 질문에 답하는 챗봇을 만드는 것이다.

핵심은 `PDF 파일 자체`를 바로 벡터 DB에 넣는 것이 아니라, PDF나 E-Book에서 추출한 `텍스트`를 잘게 나누고 임베딩해서 벡터 DB에 저장하는 것이다.

전체 흐름은 아래와 같다.

1. 메타데이터 수집 (3/18(수))
2. PDF/E-Book 원문 다운로드 (3/18(수))
3. 원문 텍스트 추출 (3/18(수))
4. 텍스트 정제 및 정규화
5. 청킹
6. 임베딩 생성
7. 벡터 DB 적재
8. 챗봇 질의 시 임베딩 기반 검색
9. 검색 결과를 LLM에 넣어 답변 생성

## 1. 메타데이터 수집

### 목적

제주 설화 API와 민담 API에서 전체 항목 목록을 받아오는 단계다.

### 입력

- `E07 제주설화정보` API
- `E08 제주민담정보` API

### 출력

각 항목의 메타데이터:

- 코드번호
- 제목
- 분류
- 핵심어
- PDF URL
- E-Book URL

### 이 단계에서 하는 일

- 페이지네이션으로 전체 항목 수집
- XML 응답 파싱
- JSONL 또는 DB 저장

### 현재 프로젝트 상태

이미 완료된 단계다.

- 설화: 182건
- 민담: 325건

## 2. PDF/E-Book 원문 다운로드

### 목적

메타데이터에 들어 있는 `pdfUrl` 또는 `ebookUrl`을 이용해 실제 원문 파일을 확보하는 단계다.

### 입력

- 메타데이터 JSONL 또는 SQLite DB

### 출력

- PDF 파일
- HTML E-Book 파일

### 이 단계에서 하는 일

- 각 항목별 원문 URL 순회
- PDF 우선 다운로드
- PDF가 없거나 실패하면 E-Book HTML 활용
- 파일 저장 및 다운로드 상태 기록

### 현재 프로젝트 상태

거의 완료된 단계다.

- 원문 파일 수집: 504개

## 3. 원문 텍스트 추출

### 목적

다운로드한 PDF 또는 HTML에서 실제 텍스트를 뽑아내는 단계다.

### 입력

- 다운로드된 PDF
- 다운로드된 HTML

### 출력

- 원문 텍스트

### 이 단계에서 하는 일

- PDF에서 텍스트 추출
- HTML 태그 제거
- 텍스트 파일로 저장
- DB에 문서 단위 저장

### 주의할 점

- PDF가 이미지 기반이면 텍스트 추출이 잘 안 될 수 있다.
- HTML E-Book은 구조가 일정하지 않을 수 있다.

### 현재 프로젝트 상태

아직 대부분 남아 있다.

- 현재 추출 완료 문서: 1건

## 4. 텍스트 정제

### 목적

텍스트 추출 결과에서 검색 품질을 방해하는 노이즈를 제거하는 단계다.

### 입력

- 추출된 원문 텍스트

### 출력

- 정제된 텍스트

### 이 단계에서 하는 일

- 반복 공백 제거
- 과도한 줄바꿈 제거
- 페이지 번호 제거
- 뷰어 노이즈 제거
- 문장 구조는 최대한 유지

### 왜 필요한가

벡터 검색은 텍스트 품질에 민감하다. 노이즈가 많으면 유사도 검색 품질이 떨어진다.

## 5. 청킹

### 목적

긴 문서를 검색 가능한 작은 단위로 쪼개는 단계다.

### 입력

- 정제된 텍스트

### 출력

- 여러 개의 텍스트 청크

### 이 단계에서 하는 일

- 문서를 일정 길이로 분할
- 청크 간 일부 중복 구간 유지
- 각 청크에 메타데이터 연결

### 권장 설정

- 청크 크기: 700~1000자
- 오버랩: 100~150자

### 왜 필요한가

문서 전체를 한 번에 임베딩하면 질문과 직접 관련 없는 내용이 섞여 검색 품질이 떨어질 수 있다. 그래서 의미 단위에 가까운 작은 조각으로 나눠야 한다.

## 6. 임베딩 생성

### 목적

각 청크를 벡터 형태로 변환하는 단계다.

### 입력

- 텍스트 청크

### 출력

- 임베딩 벡터

### 이 단계에서 하는 일

- 임베딩 모델 선택
- 각 청크를 숫자 벡터로 변환
- 질의와 같은 공간에서 비교 가능하게 만듦

### 선택해야 하는 것

- OpenAI 임베딩
- 로컬 임베딩 모델
- Hugging Face 계열 모델

### 중요 포인트

한국어 데이터이므로 한국어 성능이 충분한 임베딩 모델을 고르는 것이 중요하다.

## 7. 벡터 DB 적재

### 목적

임베딩된 청크를 빠르게 검색할 수 있도록 저장하는 단계다.

### 입력

- 임베딩 벡터
- 청크 텍스트
- 메타데이터

### 출력

- 벡터 DB 인덱스

### 이 단계에서 하는 일

- 벡터 DB 선택
- 벡터와 메타데이터 저장
- 유사도 검색 인덱스 구성

### 후보

- Qdrant
- pgvector
- Chroma
- FAISS

### 저장할 메타데이터 예시

- source_type
- code_no
- title
- category
- category2
- tags
- chunk_index

## 8. 챗봇 질의 시 임베딩 기반 검색

### 목적

사용자 질문과 관련된 청크를 벡터 DB에서 찾아오는 단계다.

### 입력

- 사용자 질문

### 출력

- 관련 청크 Top-K

### 이 단계에서 하는 일

- 질문 임베딩 생성
- 벡터 DB 유사도 검색
- 필요 시 메타데이터 필터 적용

### 예시 필터

- 설화만
- 민담만
- 특정 분류만
- 특정 제목만

## 9. 검색 결과를 LLM에 넣어 답변 생성

### 목적

검색된 텍스트 조각을 바탕으로 사용자 질문에 자연어로 답하는 단계다.

### 입력

- 사용자 질문
- 검색된 관련 청크

### 출력

- 최종 챗봇 답변

### 이 단계에서 하는 일

- 프롬프트 구성
- 검색된 문맥을 LLM에 전달
- 출처 포함 답변 생성

### 답변에 포함하면 좋은 정보

- 직접 답변
- 이야기 요약
- 출처 제목
- 설화/민담 구분
- 코드번호 또는 분류

## 이 파이프라인에서 가장 중요한 포인트

### 1. PDF 수집이 끝이라고 보면 안 된다

PDF를 모았다고 바로 챗봇이 되는 것이 아니다. 실제로 중요한 것은 PDF 안의 텍스트를 얼마나 안정적으로 추출하고 정제하느냐다.

### 2. 벡터 DB는 중간 단계다

벡터 DB는 최종 목표가 아니라 검색을 위한 저장소다. 진짜 품질은 `추출 텍스트 품질`, `청킹 방식`, `임베딩 모델`, `프롬프트 구성`에서 결정된다.

### 3. 검색과 생성은 분리해서 봐야 한다

- 검색: 관련 자료를 잘 찾는 문제
- 생성: 찾은 자료를 바탕으로 잘 설명하는 문제

둘 중 하나만 좋아도 챗봇 품질은 충분하지 않다.
